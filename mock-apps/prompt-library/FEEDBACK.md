## 4. Sharp edges

1. **Agent invocation from a slash command is mechanical, but undocumented.** `docs/PATTERNS.md` Â§3 shows how to **add** a command but never shows the syntax inside a command's markdown for actually invoking the agent â€” it just says "Invoke the spec-writer agent with $ARGUMENTS". 2. **`verify_cmd` failures in the RED phase.** In `.soup/plans/prompt-library.json` I wrote `pytest ... || true` for the RED-phase test steps so that exit-0 = "wrote the failing test, don't spawn fix-cycle." This is an established workaround but not documented. A newer user will either write `pytest ...` (triggering a spurious fix-cycle) or have no way to signal "expected red." Document the convention in `orchestrator.md` or add a `verify_expects: "fail"` field to `TaskStep`.
3. **The `Agent` tool budget isn't declared.** `orchestrator.md` line 5 lists `Agent` as one of the tools, but the model tier + prompt length budget for each dispatch isn't bounded anywhere in the schema. `budget_sec` is a wall-clock cap for the whole plan, not per-step. 4. **`/clarify` recommends amending the spec with a `## Clarifications` section** (clarify.md line 23), but `spec-writer.md` doesn't list that section in its output format. 5. **Verify-phase tooling assumptions.** My `/verify` walkthrough expected `testcontainers[postgres]` + `vitest` + a `package.json` to be present. None of these live in the mock-app until the implementer steps also create the tooling. The sequencing of "framework asks test-engineer to write a pytest that imports testcontainers" is fine, but the plan has no step for "wire up pyproject.toml / package.json / docker-compose". Either add a pre-wave bootstrap step to `/tasks`, or have the scaffolding pre-baked by a `just new prompt-library` template (which `templates/` was supposed to provide, but I didn't see a prompt-library-shaped template).
6. **Spec frozen timing.** `/specify` says specs are frozen once "referenced by an ExecutionPlan" (`specify.md` line 36). But `/plan` runs **before** `/tasks`, and `/tasks` is what produces the ExecutionPlan. So the spec is mutable during `/plan`, gets amended by `/clarify`, and only freezes after `/tasks`.
