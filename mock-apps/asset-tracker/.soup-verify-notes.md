## 2. Expected `verifier` commands

Per `.claude/commands/verify.md`, `qa-orchestrator` dispatches
`verifier` to run the plan's declared `verify_cmd`s. Each of these is
the per-step command in `mock-apps/asset-tracker/.soup/plans/asset-tracker.json`.

The most relevant end-to-end gate is run from repo root:

```bash
# 1. Restore + build (warnings as errors)
dotnet build mock-apps/asset-tracker/src/Streck.AssetTracker/Streck.AssetTracker.csproj \
  -c Debug /warnaserror

# 2. Full test suite (xUnit + FluentAssertions; in-memory EF Core)
dotnet test mock-apps/asset-tracker/tests/Streck.AssetTracker.Tests/Streck.AssetTracker.Tests.csproj \
  --nologo --logger "trx;LogFileName=results.trx" \
  --collect:"XPlat Code Coverage"

# 3. Docker-compose smoke (NFR-6)
docker compose -f mock-apps/asset-tracker/docker-compose.yml up -d --build
for i in $(seq 1 12); do
  curl -sf http://localhost:8080/health && break || sleep 5
done
docker compose -f mock-apps/asset-tracker/docker-compose.yml down -v

# 4. Plan validation
python -c "import json; from schemas.execution_plan import ExecutionPlan, ExecutionPlanValidator; p = ExecutionPlan.model_validate_json(open('mock-apps/asset-tracker/.soup/plans/asset-tracker.json').read()); ExecutionPlanValidator.from_library('library.yaml').validate(p); print('plan ok')"
```

Via the soup dotnet-wrap (`cli_wrappers/dotnet.py`):

```bash
python -m cli_wrappers.dotnet test \
  --project mock-apps/asset-tracker/tests/Streck.AssetTracker.Tests/Streck.AssetTracker.Tests.csproj
```

(That wrapper parses the TRX and emits `{counts, failures}` JSON.)

## 5. Block conditions (would trip a BLOCK verdict)

- Any `dotnet build` warning becomes an error (intentional).
- Any `#nullable disable` directive (CONSTITUTION III.5, dotnet rule §2.1).
- Any `.Result` / `.Wait()` / `.GetAwaiter().GetResult()` sync-over-async (rule §3.1).
- Missing XML doc on a public type/member (`GenerateDocumentationFile=true`).
- Migration without a sibling `.down.sql` (Constitution V.1).
- `git push --force` on main (git-ops iron law).
- Any secret in logs (Article VI.1; none expected — connection strings are config-bound).

## 6. Notes for the verifier subagent

- **Work in repo root, not `mock-apps/asset-tracker/`.** The plan's
  `verify_cmd`s are all written with paths relative to repo root so
  they match `files_allowed` globs.
- **`docker compose`** requires Docker Desktop running with WSL 2 on
  Windows (per `README.md §Windows setup`).
- **Connection string** default `Host=localhost;...` is OK for local
  `dotnet test` (uses in-memory) but prod uses
  `ConnectionStrings__Postgres` env from docker-compose.
- If the verifier hits a RED phase `verify_cmd` (those prefixed with
  `! `), exit 0 means the test correctly failed — do NOT dispatch
  the fix-cycle.
